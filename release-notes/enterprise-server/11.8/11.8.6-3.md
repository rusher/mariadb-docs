## Changes in Storage Engines

* This release incorporates MariaDB ColumnStore engine version 25.10.4

## Notable changes

* [Galera](https://app.gitbook.com/o/diTpXxF5WsbHqTReoBsS/s/3VYeeVGUV4AMqrA3zwy7/) updated to 26.4.26
* The signing key for Debian 13 "trixie" repositories has been changed, details on the new key can be found in the [MariaDB Enterprise GPG Keys](https://app.gitbook.com/s/SsmexDFPv2xG2OTyO5yV/server-management/install-and-upgrade-mariadb/installing-mariadb/binary-packages/gpg#mariadb-enterprise-gpg-keys) section of the [GPG](https://app.gitbook.com/s/SsmexDFPv2xG2OTyO5yV/server-management/install-and-upgrade-mariadb/installing-mariadb/binary-packages/gpg) page
* Sequence Fails with Galera Version 26.4.16 and later in the Presence of enabled streaming replication ([MDEV-34124](https://jira.mariadb.org/browse/MDEV-34124))
* The HashiCorp Vault key management plugin has been updated to improve availability during intermittent network or server issues. Previously, the plugin only allowed the use of cached keys if a connection timed out. This behavior has been broadened and made the default to prevent MariaDB Server interruptions. ([MDEV-38203](https://jira.mariadb.org/browse/MDEV-38203))
  * Expanded Error Handling: The plugin now treats common connectivity issues—such as “connection refused,” “no route to host,” and “internal server error”—the same as a timeout. This ensures the server continues to function using cached keys if the Vault server is briefly unreachable.
  * Updated Defaults:
    * The option to use cached values during communication errors is now enabled by default.
    * The cache timeout is now defaulting to the maximum value.
* parsec client plugin is now compiled and packaged on Windows ([MDEV-38360](https://jira.mariadb.org/browse/MDEV-38360))
* Audit log plugin will now log events for MariaDB Enterprise Cluster (Galera) replication applier operations with a generic user name: `<wsrep_applier>`. Before this change the user name in audit logs has been empty ([MDEV-35511](https://jira.mariadb.org/browse/MDEV-35511))

## Issues Fixed

### Can result in data loss

* Server crash upon moving InnoDB table with fulltext index between databases ([MDEV-31892](https://jira.mariadb.org/browse/MDEV-31892))
* When innodb\_buffer\_pool\_size is not large enough to buffer all log records during crash recovery, InnoDB may fail to recover a page from the doublewrite buffer. ([MDEV-37558](https://jira.mariadb.org/browse/MDEV-37558))
* Intermittent Replication Failure on a multi-table DELETE ([MDEV-38003](https://jira.mariadb.org/browse/MDEV-38003))
* When an inplace ALTER operation is rolled back, InnoDB drops intermediate tables and their associated FTS internal tables. However, MariaBackup's DDL tracking can incorrectly report this as a backup failure. ([MDEV-38041](https://jira.mariadb.org/browse/MDEV-38041))

* AlmaLinux 10 (x86\_64, ARM64)
* Debian 11 (x86\_64, ARM64)

* Debian 13 (x86\_64, ARM64)
* Oracle Linux 8 (x86\_64, ARM64)
* Oracle Linux 9 (x86\_64, ARM64)
* Oracle Linux 10 (x86\_64, ARM64)
* Red Hat Enterprise Linux 8 (x86\_64, ARM64)
* Red Hat Enterprise Linux 9 (x86\_64, ARM64, PPC64LE)
* Red Hat Enterprise Linux 10 (x86\_64, ARM64)

* Rocky Linux 9 (x86\_64, ARM64)
